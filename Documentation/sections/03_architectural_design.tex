\chapter{Architectural Design}
\label{chap:arch_design}
\section{Design Objective and Principles}
This chapter describes the software architecture for synthetic mixture generation, automatic speech recognition (ASR) inference under controlled identical conditions, evaluation using both established and proposed transcription metrics, and statistical analysis. The following principles guide the design:
\begin{itemize}
    \item \text{Experimental control}: overlap ratio, signal-to-noise ratio, clip duration, speaker count and noise type must be varied independently and systematically.
    \item \text{Controlled comparison}: ASR models must be evaluated under the same set of synthetic mixtures with model-specific decoding.
    \item \text{Traceability}: every result must be traceable to the specific configuration, including source files, offsets, transcripts and inference outputs.
    \item \textbf{Modularity}: all components, including mixture generation, ASR inference, metric computation and statistical analysis, should be self-contained and separable so that new models or metrics can be added without rewriting the system.
    \item \textbf{Reproducibility}: random seeds, libraries and model versions must be recorded so that other researchers can reproduce the experiment.
\end{itemize}

\subsection{Transcription Output Schema}
The output transcription JSON file has the following format:
\begin{itemize}
    \item \textbf{key}: clip ID
    \item \textbf{transcript}: a dictionary of the form \texttt{\{model name: hypothesis\}} for each ASR model, where each hypothesis is a list of tuples of the form (speaker id, text) for each utterance.
\end{itemize}

\subsection{Metric Output Schema}

The evaluation results are stored in a JSON file with three top-level fields: \texttt{summary}, \texttt{results}, and \texttt{timestamp}. 
The \texttt{summary} field contains aggregate information over the full evaluation run, including the total number of clips processed, the numbers of successful and failed evaluations, and descriptive statistics for the reported metrics. 
In particular, it stores counts and summary statistics for standard word error rate (WER), DSS-WER, and ORC-WER.

The \texttt{results} field contains a list of per-clip evaluation records. Each record corresponds to one audio clip and takes one of two forms. For an unsuccessful evaluation, the record instead stores the clip identifier, model name, evaluation status, and an error message describing the failure. 

Finally, the \texttt{timestamp} field records when the output file was generated. 
This structure separates corpus-level summary statistics from clip-level measurements, while also preserving failure information for later inspection and debugging.


% Synthetic mixture generation process
\section{Synthetic Mixture Generation}
The synthetic mixture is the core module of the system. It determines how overlap and noise are controlled. It comprises three steps: (1) \textbf{Speaker and clip selection}: randomly select the required number of speakers and clips from the clean speech dataset, (2) \textbf{Offset scheduling}:
(1) \textbf{Offset computation}: calculate the time offsets for each utterance such that the overall overlap ratio approaches the target. (2) \textbf {Residual adjustment}: adjust the time offsets calculated to ensure the overall overlap ratio exactly matches the target. (3) \textbf{Noise addition}: add noise to the mixture at the specified SNR level. The output of this process will be a set of audio with different speaker counts and overlap ratios as base clips, and each base clip will have variants with different SNR levels. The metadata and transcripts of the mixtures are stored in a manifest file.

This \emph{clip-selection} process takes a target speaker count $c$ and a duration $d$ (minimum duration of final mixed audio), and returns: (1) a sequence of clips satisfying the target speaker count and duration constraints, (2) the corresponding sequence of transcripts.

\Cref{fig:clip_selection} shows the process that is designed to create a sequence of clips with specified speaker count and duration. A set of $c$ speakers is selected at uniform random without replacement. The process iterates until the total duration of the selected clips exceeds the target duration adjusted by the overlap ratio (to account for the overlapping region that is double-counted here). The adjusted total clip duration $T_{\mathrm{src}}$ is calculated by:
\begin{equation}
T_{\mathrm{src}} = T_{\mathrm{active}} + T_{\mathrm{overlap}}.
\end{equation}
If the overlap ratio is defined as
\begin{equation}
 \mathrm{OVR} = \frac{T_{\mathrm{overlap}}}{T_{\mathrm{active}}},
\end{equation}
Then the required total source duration is 
\begin{equation}
T_\mathrm{src} = d(1+\mathrm{OVR})
\end{equation}
This formulation is preferable to the naive duration threshold because it accounts for the double-counted time by the overlapping region. Without this, a highly overlapped mixture would have a much shorter duration than the intended target.

A balanced selection is used during the clip selection process. The next speaker is chosen with a probability inversely proportional to the number of clips contributed by each speaker, to balance the number of clips per speaker (with probability 0 for speakers from the previous iteration). The random selection of clips is uniform random with replacement. This process mimics an actual conversation, with speakers taking turns.

This scheduler given a sequence of clips with durations $\ell_1, \ell_2,\dots, \ell_n$ and overlap ratio ($\mathrm{OVR} \in [0,1]$), constructs clip starting sample index such that the overlap ratio is exactly as specified, using the given overlap std ratio $\mathrm{\sigma_\mathrm{OVR}}$, and clip spacing mean and std (for non-overlapping case).

In the case where the overlap ratio is zero, the start time of each clip is determined by allocating a spacing that is a random sample from a normal distribution $\mathcal{N}(\mu$, $\mathrm{std}$) where $\mu$ and $\mathrm{std}$ are the mean and std of the clip spacing specified.
In the case where the overlap ratio is greater than zero, the target absolute overlap $o^*$ is
\begin{equation}
o^* = \frac{\mathrm{OVR}}{1+\mathrm{OVR}} \cdot \sum_{i=1}^n \ell_i.
\end{equation}
The total is distributed across the adjacent clips. Let $o_i$ be the overlap between clip $i$ and clip $i+1$, then the scheduler enforces
\begin{equation}
0\le o_i\le \min(\ell_i, \ell_{i+1}), \qquad \sum_{i=1}^{n-1} o_i=o^*.
\end{equation}
For $n$ clips, the average overlap is
\begin{equation}
\bar{o} = \frac{o^*}{n-1}
\end{equation}
then each overlap $o_i$ is sampled as
\begin{equation}
o_i=\mathcal{N}(\bar{o}, (\bar{o} \cdot \sigma_\mathrm{OVR})^2)
\end{equation}

After that, a residual correction is applied so that the sum of sampled overlaps equals the target overlap exactly. The difference $\mathrm{diff} = o^* - \sum\text{sampled overlaps}$. If the difference is greater than zero, we randomly select overlaps that are smaller than the preceding and succeeding clip durations, and add one sample to each selected overlap until the difference is zero. If the difference is negative, we randomly select overlaps greater than zero and subtract one sample until the difference is zero.
Finally, the start sample index of each clip is determined by adding the spacing (overlap will be 
treated as negative spacing). 

The start times $s_i$ are then computed recursively as
\begin{equation}
s_1 = 0, \qquad s_{i} = s_{i-1} + \ell_{i-1} - o_{i-1} \quad \text{for } i=2,3,\dots,n.
\end{equation}
An important architectural design choice here is treating the overlap as a controlled variable. This makes the generator suitable for experimental work rather than just producing plausible mixtures.

\subsection{Noise Addition}
After the clean mixture is generated, environmental noise is added.

The noise addition process takes input of the mixed audio, set of noise audio, noise type and target SNR level ($\mathrm{SNR}_{dB} \in \mathbb{R} ~\text{or}~ \text{clean}$, measured over active speech time). The noise type can be domestic (D), public (P), transportation (T), nature (N), office (O), street (S), or all (A) (see \Cref{sec:speech_noise_sources} for more details). 

Then, noise audio samples are uniformly sampled from the specified noise type(s) and concatenated until the total length is reached, with excess parts truncated. Let $A_\mathrm{active~speech}$ be the root mean square (RMS) amplitude of the active speech region, and let $A_\mathrm{noise}$ be the RMS amplitude of the noise segment before scaling. For a target SNR level of $\mathrm{SNR}_{dB}$ defined as
\begin{equation}
\mathrm{SNR}_{dB} = 20 \log_{10}\left(\frac{\text{A}_\text{active speech}}{\text{A}_\text{noise}}\right),
\end{equation}
The noise is scaled by a factor of
\begin{equation}
\alpha = 10^{-\frac{\mathrm{SNR}_{dB}}{20}} \cdot \frac{A_\text{active speech}}{A_\text{noise}}
\label{eq:noise_scaling}
\end{equation}


The active speech region is determined by the RMS of all clips in the mixture.

The final mixture is the sum of the clean mixture and the scaled noise.
\begin{equation}
x_\mathrm{final}(t) = x_\mathrm{speech}(t)+\alpha x_\mathrm{noise}(t)
\end{equation}
The SNR is measured relative to the active speech region to ensure the noise level is accurately proportional to the speech content, preventing long silence regions from distorting the effective noise level. 

This algorithm is designed to support controlled experiments on overlap ratio, SNR level, speaker count, duration and noise type by varying each factor independently. It also supports noise analysis on the same clip with different noise types and SNR levels.

\section{ASR Inference and Experiment Orchestration}
The inference layer consumes the manifest and produces standardised hypotheses for each ASR model. Each ASR backend is wrapped by a common adapter interface, enabling uniform orchestration across models. The wrapper is responsible for:
\begin{enumerate}
    \item Periodic GPU cache clearing and optional garbage collection to manage memory during large-scale inference.
    \item Skipping already transcribed clips by checking the existing JSON file
    \item Running inference over a batch of clips
    \item Serialising the output into the standardised JSON format
    \item Recording runtime and failure information
\end{enumerate}

The adapter is important because the ASR models are heterogeneous. Some return segmented outputs, while some return unsegmented outputs. Some inputs are audio waveforms, while some take file paths. This architecture, therefore, standardises the output format and provides a unified interface.
To support resumability, the wrapper checks whether a clip has already been transcribed before running inference. Exceptions are logged per batch, so a single clip failure does not break the entire process. This is practical as when large experiments are executed over days, robustness and resumability are important to prevent loss of progress. 

\subsection{Fairness Controls}
The following controls ensure that measured differences are attributed to the ASR models rather than confounding factors:
\begin{itemize}
    \item \textbf{fixed sample rate}: 16kHz for all models, resampling is applied if necessary
    \item \textbf{identical audio}: the same set of audio files is used across all models
    \item \textbf{identical reference}: the same set of reference transcriptions is used across all models
    \item \textbf{text normalisation}: the same text normalisation is applied to both reference and hypothesis before evaluation, including lowercasing and punctuation removal.
\end{itemize}



\section{Evaluation Architecture}
The evaluation system is a separate architectural block. This is where the dissertation's scientific contribution lies. 
\subsection{Established Metric Modules}
The first module computes established metrics, such as WER for single-stream referencing and ORC-WER for multi-stream referencing. 

\subsubsection{WER}
Given a reference transcript and a hypothesis transcript, WER is computed as the Levenshtein distance between the reference and hypothesis:
\begin{equation}
\mathrm{WER} = \frac{\mathrm{lev}(r,h)}{|r|}=\frac{I + D + S}{|r|}
\end{equation}
where $I$, $D$ and $S$ are the number of insertions, deletions and substitutions required to transform the hypothesis into the reference, and $|r|$ is the number of words in the reference \cite{Word_Error_Rate_Definitions}.

\subsubsection{ORC-WER}
The ORC-WER is speaker-agnostic. It computes the minimum WER over all possible segments-to-utterance assignments. It is defined as 
\begin{equation}
    \operatorname{lev}^{(\mathrm{ORC})}
=
\min_{\rho_1,\ldots,\rho_J}
\sum_{j=1}^{J}
\operatorname{lev}
\left(
\bigoplus_{u \in \rho_j} \mathcal{R}_u,\;
\bigoplus_{u' = 1}^{U_{\mathcal{H},j}} \mathcal{H}_{j,u'}
\right)
\end{equation}
where $rho_j$ contains index pairs of speaker $i$ and utterance $u$, meaning reference utterance $u$ from speaker $i$ is assigned $j$-th hypothesis segment. $\bigoplus$ is the concatenation operator.   

\subsection{Proposed Metric Module: Double-Stream Serialised WER (DSS-WER)}
The DSS-WER is a proposed metric that extends WER to solve the specific case of double-stream reference transcriptions with single-stream hypotheses. Given two reference streams $A$ and $B$ representing two speakers, and a single-stream hypothesis $H$, the DSS-WER computes the minimum WER between the hypothesis and any interleaving merging of the two reference streams. Formally, let $A$ and $B$ be a partial order of words, and let $R$ be the set of all possible interleaving mergings of $A$ and $B$ that preserve the partial order. Then the DSS-WER is defined as
\begin{equation}
\mathrm{DSS\text{-}WER}(A, B, H) = \min_{r \in R} \mathrm{WER}(r, H).
\end{equation}
\subsubsection{Example of DSS-WER vs. WER}
\Cref{fig:dss_wer_diff_exp} shows an example where the ordinary concatenated WER and the DSS-WER differ. DSS-WER is able to give zero error for a valid interleaving of the two reference streams, whereas WER penalises the different serialisation order. This illustrates the important distinction between the two metrics and the motivation for the proposed DSS-WER.

\subsection{Beam-Search Approximation for DSS-WER}
\label{sec:dss_wer_algorithm}
To evaluate a single hypothesis transcript against two reference speaker streams while allowing interleaving, I implement a beam-search approximation to the minimum levenshtein distance over all valid merges of the two reference chains. \Cref{fig:dss_wer_flow} is the overall flowchart. The full pseudocode is in \Cref{ch:dss_wer_pseudocode}.

Let
\begin{equation}
A=(a_1,a_2,\dots, a_m), \qquad B=(b_1,b_2, \dots, b_n), \qquad H=(h_1,h_2,\dots, h_T)
\end{equation}
denotes the two reference chains and the hypothesis chain.

\subsubsection{Normalisation}
Each word in the reference and hypothesis is normalised by lowercasing and removing punctuation. This is to ensure that the metric is not affected by mismatches caused by case or punctuation differences rather than recognition errors.
\subsubsection{State Definition} 
A search state is defined as
\begin{equation}
s=(i,j,t),
\end{equation}
where $i$ is the number of words consume from chain $A$, $j$ is the number of words consumed from chain $B$, and $t$ is the number of words consumed from the hypothesis $H$. The initial state is $(0,0,0)$ and the terminal states are $(m,n,T)$.
\subsubsection{Transition and Cost}
For non-terminal states $s=(i,j,t)$, there are up to 5 transitions:
\begin{enumerate}
    \item \textbf{Match / Substitution from $A$}: if $i<m$ and $t<T$, align $a_{i+1}$ with $h_{t+1}$, gives
    \begin{equation}
        (i,j,t) \to (i+1,j,t+1),
    \end{equation}
    The cost is 0 if $a_{i+1}=h_{t+1}$ and 1 otherwise.
    \item \textbf{Match / Substitution from $B$}: if $j<n$ and $t<T$, align $b_{j+1}$ with $h_{t+1}$, gives
    \begin{equation}
        (i,j,t) \to (i,j+1,t+1),
    \end{equation}
    The cost is 0 if $b_{j+1}=h_{t+1}$ and 1 otherwise.
    \item \textbf{Insertion}: if $t<T$, insert $h_{t+1}$ into the merged sequence, gives
    \begin{equation}
        (i,j,t) \to (i,j,t+1),
    \end{equation}
    The cost is 1.
    \item \textbf{Deletion from $A$}: if $i<m$, delete $a_{i+1}$, gives
    \begin{equation}
        (i,j,t) \to (i+1,j,t),
    \end{equation}
    The cost is 1.
    \item \textbf{Deletion from $B$}: if $j<n$, delete $b_{j+1}$, gives
    \begin{equation}
        (i,j,t) \to (i,j+1,t),
    \end{equation}
    The cost is 1.
\end{enumerate}
These transition preserves the partial order of the two reference chain while allowing any interleaving.
\subsubsection{Scoring Function}
Each state $s$ maintains the best exact cost $g(s)$ while candidate states are ranked using 
\begin{equation}
    f(s) = g(s) + \lambda h(s),
\end{equation} 
where $\lambda \ge 0$ is a tunable heuristic weight, and $h(s)$ is a lower bound estimate of the remaining cost.
\subsubsection{Heuristic Function}
The heuristic is the maximum of the following two estimates:
\begin{itemize}
    \item \textbf{minimum remaining edit distance}:
    \begin{equation}
    h_\mathrm{length}(s) = |((m-i) + (n-j) - (T-t))|
    \end{equation} 
    is the minimum number of insertions or deletions required to consume all remaining words from the references and hypothesis, assuming perfect matches.
    \item \textbf{local bag of words lower bound}:
    \begin{equation}
    h_\mathrm{bow}^\ell(s) = \left\lceil\frac{1}{2}\cdot\sum_{w} |\mathrm{count}(w, A_{i+1:i+\ell}) + \mathrm{count}(w, B_{j+1:j+\ell}) - \mathrm{count}(w, H_{t+1:t+\ell})|\right\rceil
    \end{equation}
    where $\mathrm{count}(w, S)$ is the number of occurrences of word $w$ in sequence $S$, and $\ell$ is the lookahead window size (truncated to the end of the sequence). This is a lower bound because it assumes that the best case where all matching words in the lookahead window can be aligned, and the remaining unmatched words can be perfectly aligned.
\end{itemize}
The heuristic is then
\begin{equation}
h(s) = \max(h_\mathrm{length}(s), h_\mathrm{bow}^\ell(s)),
\end{equation}
preserving admissibility and take the stronger approximation.
\subsubsection{Beam Pruning}
At each iteration, all successor states are ranked by $f(s)$ with ties breaker of $g(s)$. The algorithm only keeps a bounded number of candidates. The implementation has two main pruning strategies:
\begin{itemize}
    \item \textbf{adaptive beam size}: the beam size is dynamically adjusted based on the distribution of $f(s)$ scores. Given an ambiguity threshold $\tau$, base beam size $B$ and maximum beam size $B_\mathrm{max}$, if number of candidates with $f(s) \le f_\mathrm{best} + \tau$ is greater than $\frac{B}{2}$, beam size is set to $B_\mathrm{max}$, and $B$ otherwise. This allows search to be more expansive when best path is more ambiguous, and more aggressively when best path is more clear.
    \item \textbf{stratified beam pruning}: candidates are stratified into different buckets based imbalances in consumption in chain $A$ and chain $B$ (i.e. $i-j$), and a balanced number of candidates are kept from each group. The buckets are defined by $\lfloor (i-j) / \max(1, \text{bucket width})\rfloor$ (where bucket width is a tunable parameter), then the buckets are ranked by the best $f(s)$ score of each bucket, and candidates are selected in a round-robin manner from the ranked buckets until the beam size is reached. This is to prevent the beam from dominated by similar alignment and increase the diversity of the search.
\end{itemize}

\subsubsection{Bidirectional Search}
The search is run in both forward and reverse direction, and the best result from both directions are returned. The reverse direction search can discover better alignment that might be pruned in the forward direction.

\subsubsection{Final Output}
For best path distance $d^*$ found from both directions, the DSS-WER is computed as
\begin{equation}
\frac{d^*}{m+n}
\end{equation}
In addition, the best path alignment is also returned as a list of tuples of the form (operation, chain used, reference word, hypothesis word). This allows downstream error analysis and hyperparameter tunings.

\subsubsection{Complexity}
Let
\begin{itemize}
    \item $\ell$ be the lookahead window size for the heuristic,
    \item $B$ be the maximum beam size,
    \item $E$ be the maximum number of expansions (which is the upper bound of the expansions during the search)
\end{itemize}
The time complexity is
\begin{equation}
O(E(\ell + B\log B)),
\end{equation}
where there is $E$ expansions, each expansion generates up to 5 sucessors, and each successor requires $O(\ell)$ for heuristic computation and $O(B\log B)$ for sorting during beam pruning. The space complexity is $O(E)$ for storing all expanded states.

\subsection{Metric Output Schema}
All metric results are stored at clip level in a JSON file with the following fields: \texttt{clip\_id}, \texttt{model}, \texttt{wer}, \texttt{orcwer}, \texttt{dsswer}.

\section{Benchmarking}
\label{subsec:inference_time_benchmarking}

Inference-time benchmarking was used to compare the operational cost of the ASR systems under a common evaluation protocol. A fixed subset of $100$ clips was sampled from the experiment manifest using a fixed random seed, and each model was run on the same sampled set.

The runtime metric recorded was measured using \texttt{time.perf\_counter()} to capture the total wall-clock time taken for inference on the sampled set. This captures the end-to-end time instead of just the decoding time, as it reflects the actual user experience and operational cost.

The time taken per clip was computed by dividing the total runtime by the number of clips in the sampled set, while the Inverse Real-Time Factor (RTFx) was computed as:
\begin{equation}
\mathrm{RTFx} = \frac{\text{total audio duration}}{\text{Processing time}}
\end{equation}

Execution was performed on CUDA. The implementation used batch size $100$ \texttt{parakeet} as it allows batch inference, while the other systems were evaluated sequentially on the sampled clips. GPU cache clearing and garbage collection were applied between runs, but GPU memory usage was not logged as an output statistic. Likewise, failed runs were not recorded through a separate structured logging system.

\section{Real-Data Segmentation and Evaluation Pipeline}
\label{subsec:real_data_pipeline}

A separate pipeline was implemented to evaluate ASR performance on real conversational audio. Speaker-labelled utterances were first loaded from a JSON transcript, converted into timestamped intervals, and sorted chronologically. Fixed-length clips were then constructed from valid utterance boundaries so that no reference utterance was truncated across clip limits.

For each extracted clip, the corresponding waveform segment and reference transcript were written to disk as reusable evaluation artefacts. These clips were then transcribed by the selected ASR models under a common batching procedure, with intermediate outputs cached to JSON in order to support recovery from interrupted runs.

Evaluation was performed clip by clip against the exported references. Where a model returned segmented output, ORC-WER was used; otherwise, standard WER was applied to the flattened hypothesis. The pipeline therefore provided a unified framework for comparing heterogeneous ASR systems on real data while preserving both per-clip results and aggregate summary statistics.


\section{Statistical Analysis Module}
All results from transcription and metric evaluation are stored in a clip-level table indexed by clip ID and model. This table is used in the statistical analysis module.

This module converts raw clip-level scores into statistical claims about the generator and ASR models. The main functionalities include:
\begin{itemize}
    \item \textbf{Descriptive statistics}: summarise the distribution of scores under each model and condition
    \item \textbf{Paired comparisons}: perform paired statistical tests between models evaluated on the same set of clips
    \item \textbf{DSS-WER and WER comparison}: perform paired comparisons between DSS-WER and WER to evaluate the proposed metric
\end{itemize}


\section{Design Trade-offs and Limitations}
This section discusses the design trade-offs and limitations of the architectural choices.
\begin{itemize}
    \item \textbf{Controlled generation vs natural realism}: the generator is designed to support precise control over factors. This is ideal for scientific experiments but may results in less natural mixtures compared to using actual conversation.
    \item \textbf{Metric accuracy vs simplicity}: the DSS-WER approximation algorithm improves accuracy in capturing transcription errors in double-reference-single-hypothesis cases. It better reflects the transcription quality, but it is more complex and substantially more computationally expensive than WER.
\end{itemize}
